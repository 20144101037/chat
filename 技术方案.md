> 本文档基于 `chat.md` 需求编写，给出一版可落地的整体技术方案。涵盖架构设计、数据库设计、WebSocket 协议、审核流程、性能与稳定性、监控运维、API 设计、目录结构与实施计划。

## 一、方案概述

### 1.1 目标
构建一个面向投研投顾直播间场景的**多聊天室群聊系统**，核心特征：

- 支持多个独立聊天室的并行管理（创建/删除/修改、状态流转、列表查询过滤）。
- 管理员可向单个/多个聊天室广播消息；普通用户发言必须**经审核通过后**才在其已加入聊天室中展示。
- 仅支持群组交流，**不支持个人私聊**。
- 实时推送采用 WebSocket；审核为异步流程，支持超时自动拒绝。
- 满足 **1000 并发连接**、**峰值 100 条/秒** 的性能要求，并提供监控与线上问题处理流程。

### 1.2 技术选型总览

| 层次 | 技术 | 版本 | 说明 |
|------|------|------|------|
| 后端框架 | Spring Boot | 3.2.x | 核心框架，JDK 17 |
| 实时通信 | Spring WebSocket + STOMP | - | 单连接多路复用（Topic 订阅）模式 |
| 数据访问 | MyBatis / MyBatis-Plus | 3.5.x | 按需可只用原生 MyBatis |
| 主数据库 | PostgreSQL | 14+ | 事务型存储 |
| 缓存/队列 | Redis | 6+ | 会话缓存、待审核队列、Pub/Sub |
| 认证 | JWT (jjwt) | - | 无状态 Token 认证 |
| 前端 | Vue 3 + Vite | - | Element Plus + Pinia + STOMP.js |
| 监控 | Micrometer + Prometheus + Grafana | - | 指标采集与可视化 |
| 测试 | JUnit5 + Mockito + Testcontainers | - | 单测覆盖率 ≥ 80% |
| 构建 | Maven（后端） / pnpm（前端） | - | - |

### 1.3 关键设计决策

1. **WebSocket 采用 STOMP over WebSocket 的单连接多路复用**，而非"每聊天室一连接"。1000 用户若每人多房间将产生海量连接，多路复用通过 `/topic/room.{roomId}` 订阅降低连接数，更契合性能目标。
2. **审核队列使用 Redis（List + ZSet）**：List 存待审核消息 ID 保证 FIFO；ZSet 以"提交时间戳"为 score 支撑超时扫描与按提交顺序推送。消息体存 PostgreSQL，Redis 只存 ID（符合需求"缓存待审核消息 ID 列表，非消息体"）。
3. **审核服务无状态**，可水平扩展；跨实例的 WebSocket 推送通过 **Redis Pub/Sub** 广播（配合 STOMP RelayBroker 亦可，本方案默认 Redis Pub/Sub 轻量实现）。
4. **消息状态机 + 数据库乐观锁**保证审核状态更新的原子性，避免并发重复审核。
5. **推送采用 at-least-once 语义**，客户端按 `messageId` 去重，保证"不丢消息"。

## 二、系统架构设计

### 2.1 分层架构

```
┌─────────────────────────────────────────────────────────┐
│                     前端 (Vue 3 SPA)                       │
│   用户端：聊天室列表 / 聊天区 / 消息提交与状态查询            │
│   管理端：审核面板 / 聊天室管理 / 监控审计                   │
└───────────────┬───────────────────────┬───────────────────┘
        HTTP/REST(JWT)            WebSocket/STOMP(JWT)
┌───────────────▼───────────────────────▼───────────────────┐
│                   表现层 (Controller / WS)                  │
│   REST Controller  |  WebSocket 握手认证 + STOMP 端点         │
├─────────────────────────────────────────────────────────┤
│                        业务层 (Service)                     │
│  用户/认证 | 聊天室管理 | 消息服务 | 审核服务 | 会话管理 | 推送 │
├─────────────────────────────────────────────────────────┤
│                  数据访问层 (MyBatis Mapper)                 │
├──────────────────────────┬──────────────────────────────┤
│      缓存层 (Redis)        │        持久层 (PostgreSQL)       │
│  在线会话 / 待审核队列 /    │  chat_rooms / users / messages /  │
│  热门聊天室 / Pub-Sub      │  audit_logs / user_chat_rooms     │
└──────────────────────────┴──────────────────────────────┘
```

### 2.2 核心模块职责

- **表现层**：REST 接口 + WebSocket 端点；握手阶段完成 JWT 认证，STOMP 订阅时做房间权限校验。
- **业务层**
  - `UserService / AuthService`：注册、登录、Token 签发与校验、角色权限。
  - `ChatRoomService`：聊天室 CRUD、状态流转（ACTIVE/PAUSED/CLOSED）、列表过滤、成员管理。
  - `MessageService`：消息落库、历史查询、广播（管理员）。
  - `AuditService`：审核提交/通过/拒绝、批量审核、审计记录、超时处理（无状态、可扩展）。
  - `SessionManager`：在线用户会话维护（Redis），断线清理。
  - `PushService`：消息推送编排，跨实例通过 Redis Pub/Sub 广播。
- **数据访问层**：MyBatis Mapper 封装 SQL；缓存操作封装为独立 Repository。

### 2.3 部署拓扑（可水平扩展）

        Nginx / LB (支持 WebSocket 与会话保持可选)

        ▼          ▼          ▼

## 四、WebSocket 实现

### 4.1 连接模型
采用 **STOMP over WebSocket 单连接多路复用**：

- 端点：`/ws`（SockJS 兜底可选）。
- 用户全局一条连接，通过订阅不同 Topic 接收各房间消息：
  - 房间消息：`/topic/room.{roomId}`
  - 个人通知（超时/审核结果/系统通知）：`/user/queue/notifications`
  - 管理员待审核推送：`/topic/audit`（仅管理员可订阅）

### 4.2 认证与权限
- **握手认证**：`HandshakeInterceptor` 从查询参数或 `Authorization` 头取 JWT，校验通过后将 `userId/role` 存入 WS Session 属性。
- **订阅鉴权**：`ChannelInterceptor` 在 `SUBSCRIBE` 帧校验用户是否为该房间 `JOINED` 成员；`/topic/audit` 校验管理员角色。
- **发送鉴权**：用户发送 `CHAT` 只允许进入审核队列，不直接广播。

### 4.3 消息协议（JSON）

统一信封：
```json
{
  "type": "CHAT | NOTIFICATION | HEARTBEAT | SYSTEM | ACK",
  "roomId": 123,
  "messageId": 45678,
  "senderId": 1001,
  "senderName": "张三",
  "content": "文本内容",
  "status": "APPROVED",
  "timestamp": "2026-07-03T19:00:00+08:00"
}
```

| type | 方向 | 说明 |
|------|------|------|
| `CHAT` | 双向 | 聊天消息（下行为已审核通过的消息） |
| `NOTIFICATION` | 下行 | 审核结果、超时通知、加入审批结果 |
| `SYSTEM` | 下行 | 管理员系统通知/紧急消息（绕过审核） |
| `HEARTBEAT` | 双向 | STOMP 内置心跳，`heartbeat[10000,10000]` |
| `ACK` | 上行 | 客户端确认收到（用于去重与 at-least-once） |

### 4.4 断线重连与心跳
- **心跳**：STOMP heart-beat 10s；服务端 `heartbeat[10000,10000]`，超时判定失连并清理会话。
- **重连**：客户端 STOMP.js 指数退避重连（1s→2s→…→上限 30s），重连成功后**自动重新订阅**此前房间，并携带 `lastMessageId` 拉取增量历史补齐断连期间消息。
- **会话清理**：`SessionDisconnectEvent` 监听，移除 Redis 中在线会话记录。

## 五、审核流程设计

### 5.1 完整流程

```
用户提交消息
   │  ① 落库 status=PENDING_REVIEW，返回 messageId
   ▼
messages 表(PENDING_REVIEW)
   │  ② 推入 Redis：LPUSH audit:pending:queue {msgId}
   │                ZADD  audit:pending:zset {submittedAtEpoch} {msgId}
   ▼
管理员审核面板 (订阅 /topic/audit 或轮询待审核 API)
   │  ③ 通过 / 拒绝 / 批量审核
   ▼
更新 messages.status（乐观锁 version 校验）+ 写 audit_logs
   │
   ├── 通过 ④ 发布推送事件 → PushService
   │        按 submitted_at 顺序 → /topic/room.{roomId} 广播 CHAT
   │        从 Redis 队列移除该 msgId
   │
   └── 拒绝    从队列移除，通知提交者 (NOTIFICATION)

后台定时任务（每 5s）扫描 audit:pending:zset
   │  ⑤ score < now - maxWaitSeconds 的消息
   ▼
批量更新 status=TIMEOUT + audit_logs(action=TIMEOUT, reviewer=NULL)
   │  从队列移除 → 通知提交者"消息已超时被拒绝"
   ▼ 超时消息不进入聊天室
```

### 5.2 数据一致性保障
- **状态更新原子性**：`UPDATE messages SET status=?, version=version+1 WHERE id=? AND version=? AND status='PENDING_REVIEW'`，`affected rows=0` 表示已被他人处理（并发/超时抢占），拒绝重复操作。可选叠加 Redis 分布式锁 `SETNX audit:lock:{msgId}`。
- **推送顺序**：以 `submitted_at`（同房间必要时用 `id` 兜底）排序，保证"按消息提交时间顺序"展示。
- **at-least-once**：推送后不立即认为成功；客户端以 `messageId` 去重，重连拉历史补齐。避免消息丢失。
- **审核与推送解耦**：审核事务提交后再发推送事件（`@TransactionalEventListener(AFTER_COMMIT)`），防止事务回滚导致误推。

### 5.3 超时任务多实例安全
定时任务用 **ShedLock**（基于 PostgreSQL/Redis 锁）保证多实例部署下同一时刻仅一个实例执行超时扫描。

## 六、性能与稳定性

| 需求 | 方案 |
|------|------|
| 1000 并发连接 | STOMP 单连接多路复用；调优 `server.tomcat.threads.max`、WS buffer；单机可承载，多实例横向扩展 |
| 峰值 100 条/秒 | 审核/推送异步化（线程池 + 事件驱动）；Redis 队列削峰；批量审核减少 DB 往返 |
| 容错 | WS 自动重连；HikariCP 连接池（`maximum-pool-size` 按实例数 × 20 估算）；审核服务无状态 |
| 事务性 | 审核状态更新走 DB 事务 + 乐观锁；成员列表等最终一致（可接受秒级延迟，通过事件同步） |
| 降级 | 审核队列积压超阈值时告警；Redis 不可用时消息落库仍可保存，审核降级为纯 DB 轮询 |

**压测方案（可选）**：使用 JMeter/Gatling 模拟 1000 WS 连接 + 100 msg/s 提交与审核，观测 P95/P99 延迟、队列长度、错误率，定位瓶颈并验证降级预案。

## 七、监控与线上问题处理流程

### 7.1 监控指标（Micrometer → Prometheus → Grafana）
- 在线用户数 / 活跃连接数（自定义 Gauge）
- 消息处理延迟 P95/P99（Timer：提交→推送）
- 审核队列长度（Gauge，读取 Redis 队列 size）
- 系统错误率、接口 QPS/延迟（Spring Boot Actuator）
- 数据库连接池使用率（HikariCP metrics）、Redis 命中率

### 7.2 线上问题处理流程
1. **日志采集**：统一 Logback JSON 结构化日志 + TraceId（Sleuth/Micrometer Tracing），可接入 ELK。
2. **指标报警**：Prometheus Alertmanager 规则——队列积压 > 阈值、P99 延迟 > 阈值、错误率 > 1%、连接池耗尽即告警（企业微信/邮件）。
3. **故障排查**：按 TraceId 串联请求链路；Actuator `/health`、`/metrics` 快速定位；Grafana 大盘观测趋势。
4. **回滚预案**：镜像/制品版本化，保留上一稳定版本；数据库变更用 Flyway 且提供回滚脚本；灰度发布，异常一键回滚。

## 九、前端设计（Vue 3）

- **技术栈**：Vue 3 + Vite + Pinia + Vue Router + Element Plus + @stomp/stompjs + axios。
- **用户端页面**：登录/注册、聊天室列表（过滤/加入）、聊天室详情（实时消息区 + 输入框 + "待审核"状态提示）、我的消息（审核状态）。
