# 架构设计优化说明

> 本文档基于当前已实现代码（`com.jin.chat`），阐述在原始需求之上的架构优化与工程化取舍。与 [`技术方案.md`](../技术方案.md) 互为补充：技术方案偏总体设计，本文档偏**落地后的优化细节**。

---

## 一、总体架构

```
┌──────────────────────────────────────────────────────────────────┐
│                     前端 Vue 3 SPA（Pinia + Router）               │
│  工作台 / 聊天室 / 审核 / 广播 / 监控 / 全局配置 / RBAC 系统管理      │
└───────────────┬────────────────────────────┬─────────────────────┘
        HTTP/REST (JWT)                WebSocket/STOMP (JWT)
┌───────────────▼────────────────────────────▼─────────────────────┐
│              表现层：REST Controller + STOMP 端点                   │
│  JwtAuthInterceptor（REST）  WebSocketAuthInterceptor + StompChannelInterceptor │
├──────────────────────────────────────────────────────────────────┤
│  业务层：Auth / ChatRoom / Member / Message / Audit / Push / RBAC … │
├──────────────────────────────────────────────────────────────────┤
│  数据访问：MyBatis-Plus Mapper                                      │
├──────────────────────────┬───────────────────────────────────────┤
│  Redis（队列/会话/缓存）    │  MySQL / PostgreSQL（Flyway 迁移）      │
└──────────────────────────┴───────────────────────────────────────┘
```

## 二、核心优化点

### 2.1 WebSocket：单连接多路复用 + 全局单例

| 问题 | 优化 |
|------|------|
| 各页面独立建连/断连，切换路由导致推送中断 | 前端 `stores/ws.js` 在 `Home.vue` 登录后建立**全局单例连接**，子页面仅订阅/取消 Topic |
| `SessionConnectedEvent` 时 Principal 未就绪，在线标记丢失 | 在 `StompChannelInterceptor` 的 **CONNECT** 帧写入 Redis 会话；SUBSCRIBE 仅对敏感 Topic 鉴权 |
| 订阅 `/user/queue/notifications` 时 `getUser()` 为空导致连接被误杀 | SUBSCRIBE 鉴权改为「仅校验 `/topic/room.*` 与 `/topic/audit`」，个人通知队列不做拦截 |

**订阅目的地**

| 目的地 | 用途 | 鉴权 |
|--------|------|------|
| `/topic/room.{roomId}` | 房间已通过审核的消息 | 须为 `JOINED` 成员 |
| `/user/queue/notifications` | 审核结果、超时、系统通知 | 已认证即可 |
| `/topic/audit` | 新待审核消息提醒 | 须拥有菜单 `/app/audit` |

**可靠性**：推送 at-least-once + 客户端 `messageId` 去重；断线 STOMP 自动重连并 `resubscribeAll()`；历史消息 REST 补齐。

---

### 2.2 在线人数统计：登录态而非 WebSocket 连接数

| 维度 | 说明 |
|------|------|
| 统计口径 | **登录成功且未退出/未过期**的用户数（Redis `chat:auth:logged-in:{userId}`） |
| 写入时机 | `POST /api/auth/login` 成功 |
| 续期时机 | 携带有效 JWT 的 REST 请求、WebSocket 握手成功 |
| 清除时机 | `POST /api/auth/logout`、重置密码（`TokenValidityRepository` + 清除登录态） |
| TTL | 与全局配置 `jwt.expire-minutes` 一致（默认 120 分钟） |
| 展示名称 | 监控/工作台界面统一展示为「**在线人数**」 |

WebSocket 会话（`chat:session:online:*`）仍单独维护，用于推送链路，**不参与**在线人数指标。

---

### 2.3 审核链路：事务 + 异步推送 + 并发安全

```
用户提交 → messages(PENDING_REVIEW) → Redis 待审核队列
    → 管理员审核（乐观锁 CAS + 分布式锁）
    → TransactionUtils.afterCommit → PushService 异步推送
    → /topic/room.{roomId}
```

| 机制 | 作用 |
|------|------|
| `messages.version` + `updateStatusCas` | 并发审核/超时只生效一次 |
| `DistributedLock`（`chat:audit:lock:{messageId}`） | 单条消息审核互斥 |
| `TransactionUtils.afterCommit` | 事务提交后再推送，避免幽灵消息 |
| `@Async("pushExecutor")` | 审核与推送解耦、削峰 |
| Redis List + ZSet | List FIFO 展示；ZSet 按提交时间扫描超时 |

---

### 2.4 RBAC：角色-菜单 + 前后端双端鉴权

原始方案仅用 `users.role` 粗粒度区分；现扩展为完整 RBAC：

- **数据模型**：`roles` / `menus` / `user_roles` / `role_menus`
- **后端**：`MenuPermissionService.requireMenuPath()` 按**菜单路径**（如 `/app/audit`）校验，支持自定义角色（如「业务管理员」）灵活授权
- **前端**：`GET /api/me/menus` 动态侧边栏 + `router.beforeEach` 路由守卫 + `canAccessRoute()`
- **缓存**：`PermissionCache` + 二级缓存，角色/菜单变更后 Pub/Sub 跨实例失效

保留 `LoginUser.role` 主角色字段，用于部分 `isAdmin()` 快捷判断（聊天室 CRUD、成员管理等）。

---

### 2.5 二级缓存（Guava L1 + Redis L2）

高并发读路径（权限、全局配置）采用 `TwoLevelCache`：

| 层级 | 说明 |
|------|------|
| L1 | 进程内 Guava Cache，带单飞（single-flight）防击穿 |
| L2 | Redis 共享缓存 |
| 失效 | 写路径提交后发布 `chat:cache:invalidate` 频道，各实例清除 L1 |

接入点：`PermissionCache`（用户角色/菜单）、`SysConfigService`（配置项如 `jwt.expire-minutes`）。

---

### 2.6 JWT 无状态 + 强制失效

| 能力 | 实现 |
|------|------|
| 签发 | `JwtUtil`，有效期读 `system_config.jwt.expire-minutes` |
| 校验 | REST / WebSocket 统一解析 JWT |
| 重置密码强制下线 | `TokenValidityRepository` 记录 `minIat`，早于该时间的 Token 一律无效 |
| 主动退出 | `POST /api/auth/logout` 清除登录态 Redis |

---

### 2.7 全局配置收敛到数据库

业务参数（审核超时、消息长度、JWT 有效期等）写入 `system_config`，运行期可在「全局配置」菜单修改；`jwt.expire-minutes` 为**内置只读项**（不可删、不可通过 UI 增删配置项，改值有校验）。

---

### 2.8 数据库双 vendor 支持

通过 Spring Profile + Flyway `{vendor}` 占位符同时支持 **MySQL**（默认，本地测试）与 **PostgreSQL**（生产推荐）：

- 配置：`application-mysql.yml` / `application-postgres.yml`
- 迁移：`db/migration/mysql/` 与 `db/migration/postgresql/`
- MyBatis-Plus 分页插件自动识别方言

---

### 2.9 聊天室成员管理增强

在原始「加入/审批/退出」基础上增加：

- **人员管理**：列表查询、直接拉人（`add`）、踢人（`kick`）、候选人搜索（`member-candidates`）
- 状态变更并入聊天室「修改」弹窗；列表页「状态」列改为「人员管理」入口

---

### 2.10 监控与可观测性

| 指标 | 来源 |
|------|------|
| 在线人数 | `LoggedInRepository.loggedInCount()` |
| 待审核队列长度 | Redis 队列 size |
| 审核延迟 P95/P99 | Micrometer Timer |
| 错误率 / HTTP 请求 | Actuator + 自定义 Counter |
| 连接池 | HikariCP Gauge |

暴露：`GET /api/metrics/dashboard`（需菜单权限）、`/actuator/prometheus`。

## 三、扩展与部署建议

| 场景 | 建议 |
|------|------|
| 多实例 WebSocket 推送 | 接入 Redis Pub/Sub 或 STOMP Relay（RabbitMQ），各实例订阅后本地 `SimpMessagingTemplate` 推送 |
| 审核超时任务多实例 | ShedLock 保证单实例扫描 |
| Redis `KEYS` 统计在线人数 | 生产可改为 Redis Set `chat:auth:logged-in:users` + SCARD |
| 压测目标 | 1000 WS 连接、100 msg/s 提交，观测队列积压与 P99 |

---

## 四、前端架构要点

- **布局**：`Home.vue` 侧栏 + 顶栏，子路由切换不销毁 WS
- **状态**：Pinia（`auth` / `ws`）
- **API**：axios 统一拦截 JWT 与错误处理
- **主题**：`theme.css` + Element Plus 定制
- **权限**：菜单驱动路由，无权限访问重定向工作台
