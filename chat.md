# 笔试题目
## 一、背景
在投研投顾业务沟通场景中，存在对多聊天室（面向直播间聊天场景）并行管理、消息审核与群组交流的特定需求。本题目要求设计一个支持多聊天室架构的群聊系统，核心功能包括：管理员可向指定聊天室发送消息，用户发言需经审核后方可显示，系统不支持个人私聊仅支持群组交流。该设计旨在考察候选人在实时通信、权限控制与系统稳定性方面的综合能力，特别是AI编程辅助在复杂业务场景（如异步审核、实时推送）中的实际落地与整合能力。

## 二、核心功能要求
### 1. 聊天室管理
+ 支持创建、删除、修改多个独立聊天室
+ 每个聊天室具备唯一标识符和可配置属性（名称、描述、最大用户数）
+ 聊天室状态管理（活跃、暂停、关闭）
+ **用户加入/退出聊天室流程（详见5.用户功能）**
+ 聊天室列表查询与过滤功能

### 2. 消息发送与接收
+ 管理员可向单个或多个聊天室广播消息
+ **用户提交消息至审核队列，审核通过后在其已加入的对应聊天室中显示**
+ 实时消息推送采用WebSocket协议
+ 消息格式标准化（文本、时间戳、发送者ID、聊天室ID、消息状态）
+ 消息历史记录存储与查询

### 3. 审核机制
+ 所有普通用户发言必须经过管理员审核
+ 审核队列管理（待审核、已通过、已拒绝、已超时）
+ 审核操作记录与审计追踪
+ 支持批量审核操作
+ 审核时效性要求：配置最大审核等待时间（如30秒）
+ **审核超时处理：消息在队列中超过最大等待时间后，自动标记为“已超时”并拒绝，通过系统通知告知提交用户。超时消息不进入聊天室。**

### 4. 管理员功能
+ 管理员身份认证与权限分级（如系统管理员、聊天室管理员）
+ 聊天室创建与管理权限
+ 消息审核操作界面（支持按聊天室、状态筛选）
+ 系统监控与用户行为审计
+ 紧急消息发送权限（可绕过审核流程，直接向聊天室发送系统通知）

### 5. 用户功能
+ 用户注册与身份验证
+ 聊天室浏览与加入申请（用户可申请加入开放或需要审批的聊天室）
+ **用户加入/退出聊天室流程：**
    - **加入：** 用户浏览聊天室列表 -> 选择目标聊天室 -> 发起加入请求（若聊天室为开放状态则直接加入；若为审批状态，则等待管理员批准） -> 加入成功后，建立与该聊天室的WebSocket连接，开始接收消息。
    - **退出：** 用户主动退出特定聊天室 -> 系统断开该聊天室的专属WebSocket连接 -> 更新用户-聊天室关联状态。
+ 消息提交与审核状态查询
+ 已审核消息实时查看
+ 个人消息历史记录查询

## 三、详细设计要求
### 1. 系统架构设计
采用分层架构：

+ **表现层：** WebSocket连接管理。
+ **业务层：** 消息处理、审核逻辑、聊天室管理、用户会话管理。
+ **数据访问层：** 封装对数据库与缓存的操作。
+ **缓存层：** 使用Redis缓存在线用户会话、热门聊天室信息、待审核消息ID列表（非消息体）等。
+ **框架：** Spring Boot作为核心框架，集成WebSocket实现实时通信。
+ **监控与运维（可选）：** 集成监控组件（如Prometheus）实时追踪系统健康状态，包括连接数、消息延迟、审核队列积压等核心指标。**需设计线上问题处理流程，包括日志采集、指标报警、故障排查与回滚预案。**

### 2. 数据库设计
PostgreSQL作为主数据库，设计以下核心表结构：

+ **chat_rooms** `聊天室表`
+ **users** `用户表`
+ **messages** `消息表`
+ **audit_logs** `审核日志表`
+ **user_chat_rooms** `用户-聊天室关联`

`可根据自己的理解进行增减或修改，能实现需求即可；`

### 3. WebSocket实现
基于WebSocket实现全双工通信，设计要点包括：

+ **连接管理：** 为每个用户到每个加入的聊天室建立独立的WebSocket会话，或使用单连接多路复用（Topic订阅）模式。
+ **消息协议：** 定义JSON格式的通信协议，包含`type`（如：`CHAT`, `NOTIFICATION`, `HEARTBEAT`）、`payload`等字段。
+ **断线重连：** 客户端实现自动重连机制，并在重连后恢复会话状态（如重新订阅聊天室）。
+ **心跳机制：** 建立心跳包机制（Ping/Pong）以检测连接健康度，及时清理失效连接。
+ **安全性：** 连接建立时需进行身份认证（如携带Token），并对发送的消息进行权限校验。

### 4. 审核流程设计
1. **提交：** 用户提交消息 -> 消息以`PENDING_REVIEW`状态存入`messages`表 -> 事件触发，将消息ID推入待审核队列（如Redis List或消息队列）。
2. **展示：** 管理员界面轮询或WebSocket接收待审核列表（包含消息内容、提交者、聊天室、提交时间）。
3. **审核：** 管理员执行操作（通过/拒绝） -> 更新`messages`表状态，并记录`audit_logs` -> 若通过，触发消息推送事件。
4. **推送：** 业务层查询消息完整信息，**按消息提交时间顺序**，通过WebSocket推送至对应聊天室所有在线用户。
5. **超时：** 后台定时任务扫描超时消息，自动更新状态为`TIMEOUT`并记录日志，同时通知用户（可通过站内信或专用通知WebSocket通道）。
6. **数据一致性保障：** 审核与推送过程需考虑并发操作，可通过数据库乐观锁或分布式锁确保状态更新的原子性。消息推送应采用至少一次（at-least-once）语义，避免消息丢失。

### 5. 性能与稳定性
+ **连接数预估：** 支持至少**1000个**并发WebSocket连接。
+ **消息吞吐：** 峰值每秒处理**100条**消息（含用户提交与系统推送）。
+ **容错机制：** WebSocket连接异常自动恢复，数据库连接池合理配置，关键服务（如审核服务）无状态化设计以支持水平扩展。
+ **数据一致性保障：** 关键状态更新（如消息审核）需保证事务性；最终一致性场景（如聊天室成员列表同步）需明确同步机制与可接受延迟。
+ **监控指标（可选）：** 在线用户数、消息处理延迟（P95/P99）、审核队列长度、系统错误率、数据库连接池使用率。
+ **压力测试（可选）：** 模拟高并发场景（如短时间内大量用户加入、消息提交、审核操作），验证系统瓶颈与降级预案。

## 四、技术要求
### 1. 后端技术栈
Spring Boot 3.x，集成WebSocket，数据库访问层使用MyBatis。

### 2. 前端要求
Vue.js 3或React 18实现管理界面与用户界面，支持WebSocket连接，界面需清晰展示聊天室列表、消息审核面板、实时聊天区域。

### 3. 数据库要求
PostgreSQL 14+。

### 4. 缓存要求
Redis 6+。

### 5. 部署要求
无要求，实现需求即可。
